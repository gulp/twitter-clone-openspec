### 1) Executive Assessment

**System Maturity: Prototype Concept, Critical Production Vulnerabilities.**
The provided `IMPLEMENTATION_PLAN.md` demonstrates a cohesive local-development mental model but catastrophically fails under distributed, high-concurrency production scenarios. It relies on in-memory state, synchronous side-effects, and naive caching strategies that will result in state drift, resource exhaustion, and data leaks during network partitions or traffic spikes.

**Top 3 Blind Spots:**

1.  **Single-Node Real-Time Delusion:** The plan explicitly mandates an in-process `EventEmitter` for SSE. In a multi-server or serverless environment, events will only reach users connected to the exact same physical container where the event originated.
2.  **Synchronous Side-Effect Blocking:** Performing network I/O (SMTP email sending) within the primary web request thread will exhaust connection pools and timeout downstream requests during latency spikes.
3.  **The Thundering Herd Cache Collapse:** A hard 60-second TTL on the home feed without a Stale-While-Revalidate (SWR) mechanism or lock will cause massive concurrent database hits the millisecond the cache expires for highly active users.

### 3) Proposed Revisions

#### Revision 1: Single-Node SSE State

**Title:** Eradicate In-Process Event Emitters
**Category:** Scalability
**Severity:** High
**The Weakness:** The plan uses Node.js `EventEmitter` for SSE. If User A is connected to Server 1, and User B likes their tweet on Server 2, Server 1 never gets the event.
**The Fix:** Mandate Redis Pub/Sub for cross-node event distribution.

```diff
- ### 1.8 SSE Implementation
- The spec says "single multiplexed SSE connection." Decision: a Next.js Route Handler at `GET /api/sse` that checks auth, holds the connection open, and writes `text/event-stream` responses. Server-side publishing uses an in-process `EventEmitter` (single-server assumption for v1). Event types: `new-tweet`, `notification`. For multi-server, swap EventEmitter for Redis Pub/Sub — the abstraction (`src/server/services/sse-publisher.ts`) supports both.
+ ### 1.8 SSE Implementation
+ The spec says "single multiplexed SSE connection." Decision: a Next.js Route Handler at `GET /api/sse` that checks auth, holds the connection open, and writes `text/event-stream` responses. Server-side publishing MUST use Redis Pub/Sub. In-process EventEmitters are strictly forbidden to ensure multi-node horizontal scalability. Event types: `new-tweet`, `notification`.
```

#### Revision 2: Synchronous Side-Effects

**Title:** Asynchronous Background Workers for I/O
**Category:** Performance / Resilience
**Severity:** High
**The Weakness:** Sending password reset emails directly in the tRPC mutation blocks the request thread. If the SMTP server is slow, the user request times out, but the email might still send, leading to retries and spam.
**The Fix:** Introduce a background job queue (e.g., BullMQ) backed by Redis for all outbound communication.

```diff
- | `src/server/services/email.ts` | sendPasswordResetEmail using nodemailer |
+ | `src/server/workers/email.worker.ts` | BullMQ worker processor for async email jobs |
+ | `src/server/services/email.ts` | Enqueues jobs to Redis `email-queue` (does not send directly) |
```

#### Revision 3: The Write-Lag Problem

**Title:** Decouple Search Indexing from Core Writes
**Category:** Performance
**Severity:** Medium
**The Weakness:** Postgres generated columns for TSVECTOR block the insert/update transaction. For high write volumes, this degrades DB performance.
**The Fix:** Offload search indexing to a background queue or logical replication stream. _Note: For V1, we will patch the plan to acknowledge this risk and use a background job for non-critical indexing._

```diff
- PostgreSQL `tsvector` column on Tweet with a GIN index. Use a generated column `searchVector tsvector GENERATED ALWAYS AS (to_tsvector('english', content)) STORED`. This avoids triggers and keeps the schema self-contained.
+ PostgreSQL `tsvector` column on Tweet with a GIN index. Do NOT use `GENERATED ALWAYS`. Tweet creation enqueues a `search-index` background job in Redis, which asynchronously updates the `searchVector` to prevent blocking the primary write transaction.
```

#### Revision 4: Blind Media Trust

**Title:** Media Chain of Trust Implementation
**Category:** Security
**Severity:** High
**The Weakness:** Client requests an upload URL, uploads the file, and then passes arbitrary S3 URLs to the `/tweet` endpoint. The server has no cryptographic proof the client actually uploaded to the authorized path.
**The Fix:** Implement an "Upload Intent" flow. The DB tracks pending uploads.

```diff
- | `src/server/trpc/routers/media.ts` | getUploadUrl mutation |
+ | `src/server/trpc/routers/media.ts` | requestUploadIntent, finalizeUpload |
+
+ **Key details:**
+ - `requestUploadIntent`: Create `MediaRecord` in DB (status: PENDING), return presigned URL with strict Content-Length constraints.
+ - `finalizeUpload`: Server verifies S3 object metadata. Updates `MediaRecord` to UPLOADED. Tweets can only attach UPLOADED `MediaRecord` IDs, not raw URLs.
```

#### Revision 5: Observability Black Hole

**Title:** Mandatory Telemetry and Health Probes
**Category:** Observability
**Severity:** High
**The Weakness:** The system cannot be debugged in production. There are no request IDs tying logs together, nor health checks for orchestrators (Kubernetes/Docker Swarm) to cycle dead containers.
**The Fix:** Introduce Pino for structured JSON logging, middleware for correlation IDs, and specific Liveness/Readiness probes.

```diff
- | `src/app/api/trpc/[trpc]/route.ts` | tRPC HTTP handler |
+ | `src/app/api/trpc/[trpc]/route.ts` | tRPC HTTP handler (Wrapped with RequestId and Pino Logger context) |
+ | `src/app/api/health/route.ts` | Readiness/Liveness probe returning DB/Redis connection status |
```

#### Revision 6: Lack of Write Idempotency

**Title:** Idempotency Keys for Mutations
**Category:** Scalability / Data Integrity
**Severity:** Medium
**The Weakness:** If a user clicks "Tweet" on a train and their connection drops before the 200 OK arrives, they will click it again. The server will create two identical tweets.
**The Fix:** Require an `Idempotency-Key` header/payload for state-changing operations.

```diff
- - `create`: validate content length, require text or media, parse mentions, create tweet in transaction
+ - `create`: Requires client-generated UUID as `idempotencyKey`. Server checks Redis `idemp:user:{userId}:{key}` before processing. If exists, return cached response. Otherwise, process, create tweet in transaction, and cache response.
```

#### Revision 7: Database Counter Race Conditions

**Title:** Atomic Increments Only
**Category:** Consistency
**Severity:** High
**The Weakness:** The phrasing "conditionally increment (only if new)" implies querying the DB, calculating count + 1 in memory, and writing it back. Under concurrency, counts will drift.
**The Fix:** Enforce Prisma's `increment` and `decrement` atomic operators.

```diff
- - `like`: upsert Like, conditionally increment likeCount (only if new), fire like notification
+ - `like`: upsert Like. If operation was an insert, use Prisma `update` with `{ likeCount: { increment: 1 } }` atomically in the same transaction. Never calculate counts in memory.
```

#### Revision 8: Soft-Delete Cache Leakage

**Title:** Immediate Cache Invalidation on Soft Deletes
**Category:** Consistency
**Severity:** Medium
**The Weakness:** A user deletes an inappropriate tweet. The DB marks it `deleted = true`. However, the Redis cached home feed (`feed:{userId}:page:{cursor}`) still contains the tweet for 60 seconds.
**The Fix:** Soft deletes must forcibly scan and prune or immediately evict relevant user feed caches.

```diff
- - `delete`: verify authorId === session.userId, set deleted=true, decrement tweetCount
+ - `delete`: verify authorId === session.userId, set deleted=true, decrement tweetCount. Emit Redis Pub/Sub event to forcibly invalidate `feed:*:page:*` caches containing this tweet ID, or invalidate the author's followers' cached feeds.
```

#### Revision 9: Account Enumeration Leak

**Title:** Constant-Time, Opaque Auth Responses
**Category:** Security
**Severity:** Low
**The Weakness:** "Duplicate email/username shows correct error messages." This allows attackers to test if an email is registered to a sensitive account.
**The Fix:** Registration and password resets must return generic success messages to unauthenticated users.

```diff
- - Duplicate email/username shows correct error messages
+ - Duplicate email registration returns success (but sends a "You are already registered" email via background worker). API must not leak account existence.
```

#### Revision 10: State Drift (Cache vs DB)

**Title:** Transactional Outbox for Cache Busting
**Category:** Consistency
**Severity:** Medium
**The Weakness:** Setting `feed:{userId}:invalidate` flag in Redis _after_ DB commit can fail if the Node process crashes between the two steps. The DB is updated, but the cache is never busted.
**The Fix:** Use a basic outbox pattern or acknowledge the drift window.

```diff
- | `feed:{userId}:invalidate` | String | 5s | Cache-bust flag on new follow/tweet |
+ | `feed:{userId}:invalidate` | String | 5s | Cache-bust flag. Set via Redis Lua script to guarantee atomicity with other Redis state changes, failing open (cache miss) on connection errors.
```

#### Revision 11: The Thundering Herd Cache Collapse

**Title:** Stale-While-Revalidate and Cache Jitter
**Category:** Scalability
**Severity:** High
**The Weakness:** A 60s hard TTL means at second 61, 500 concurrent requests from a user's followers will bypass the cache and hit the database simultaneously.
**The Fix:** Implement background cache refresh (SWR).

```diff
- 6. Cache result in Redis
+ 6. Cache result in Redis. Implement Stale-While-Revalidate: return stale cache if within TTL + Jitter window, but asynchronously trigger a single DB query to refresh the cache. Use Redis `SETNX` lock to ensure only one process rebuilds the cache.
```

#### Revision 12: OAuth Username Collisions

**Title:** Deterministic Username Retry Handling
**Category:** Consistency
**Severity:** Low
**The Weakness:** "Append random 4-digit suffix. If collision, retry". In high concurrency, doing a read-then-write retry loop fails.
**The Fix:** Rely on DB constraints and catch the unique constraint violation code.

```diff
- **v1 decision:** Auto-generate username... append random 4-digit suffix if collision.
+ **v1 decision:** Auto-generate username. Attempt insert. If Prisma throws `P2002` (Unique constraint failed), catch, generate new suffix, and retry (max 3). Do not check existence before inserting.
```

### 5) Prioritization

**Must Do Now (Foundational)**

- **Revision 1 (Redis Pub/Sub):** If you build the app around `EventEmitter`, breaking it apart later for multiple nodes requires rewriting the entire real-time layer.
- **Revision 7 (Atomic Increments):** Do this from day one. Data corruption via race conditions is impossible to fix retroactively.
- **Revision 4 (Upload Intent):** You cannot launch with an open S3 bucket configuration that trusts arbitrary strings.

**Should Do Soon (Scaling & Ops)**

- **Revision 2 (Background Workers):** Essential before real users start triggering password resets, or your API will lock up.
- **Revision 5 (Observability):** Without logs and RequestIds, your first production outage will be entirely blind.
- **Revision 11 (SWR Cache):** Necessary before traffic hits, otherwise the DB will fall over during your first minor spike.

**Nice to Have Later (Optimization)**

- **Revision 3 (Async Search Indexing):** Postgres generated columns will hold up fine for MVP traffic; move to async when write latency becomes an issue.
- **Revision 6 (Idempotency Keys):** Important for mobile clients on spotty networks, but the core app functions without them initially.
