## 1) Executive Assessment

**System Maturity:** Mid-Level Concept with Critical Production Vulnerabilities.
The baseline architecture exhibits a reasonable grasp of modern web patterns (cursor pagination, Redis caching, denormalized counts). However, it relies heavily on "happy path" assumptions that will catastrophically fail under high concurrency, distributed network partitions, and adversarial load. The design operates under the illusion that synchronous operations and simplistic locks are sufficient for a system modeled around viral, exponential data graphs (Twitter).

**Blind Spots (The 3 Most Dangerous Assumptions):**

1.  **Synchronous Side Effects & Timing Attacks:** The plan assumes SMTP operations (password reset, registration emails) can occur inline with the HTTP request. Even if the logic "always returns a generic success message," the TCP handshake and TLS negotiation with an SMTP server take 500ms to 3s. An attacker measuring request latency will trivially enumerate your entire user base. If the email exists, the request takes 2 seconds; if not, it takes 50ms.
2.  **The Mass Invalidation Thundering Herd:** The feed caching strategy relies on a monotonic `feed:version:{userId}` that bumps on _any_ follow, unfollow, retweet, or delete. If a user with 5 million followers deletes a tweet, the system synchronously attempts to bust 5 million cache versions. The subsequent read requests from those 5 million users will instantly bypass the cache, hit the database via the SETNX lock, and exhaust the PostgreSQL connection pool, triggering a cascading total system failure.
3.  **Untrusted Media State:** The architecture trusts client-provided S3 URLs simply by matching string prefixes during tweet creation. An attacker can request a presigned URL, upload a 5MB payload of malware or illegal content, and then never attach it to a tweet, creating uncollectible, unmoderated garbage in your S3 bucket. Furthermore, they can attach _other_ users' public media URLs to their own tweets, bypassing origin tracking.

## 3) Deep-Dive Architectural Revisions

**Title:** Synchronous SMTP Timing Attack & Thread Blocking
**Category & Severity:** Security & Performance - High
**The Failure Mode:** The plan executes `nodemailer` send operations during the `requestReset` and `register` tRPC mutations. If the system checks `user exists` and then calls SMTP, the 1-3 second network latency to the SMTP relay acts as an oracle. Attackers can perform timing attacks to verify account existence, bypassing the "generic success message" mitigation. Furthermore, under high load, awaiting external SMTP servers will exhaust the Node.js event loop and backend connection pools.
**The Architectural Fix:** Implement a distributed task queue (e.g., BullMQ backed by Redis). The HTTP handler must strictly validate input, push a `{ type: 'RESET_EMAIL', email, token }` job to the queue, and return 200 OK in under 50ms. Background worker nodes process the queue asynchronously, absorbing SMTP latency and providing exponential backoff/retry guarantees without impacting the client-facing API.

**Title:** Unverifiable Media Chain of Trust (Orphan Uploads)
**Category & Severity:** Security & Cost - High
**The Failure Mode:** The plan hands out Presigned URLs and trusts the client to pass the URL back during `createTweet`, relying only on string prefix validation. Clients can request URLs and upload files without ever linking them to a tweet, resulting in terabytes of orphaned S3 objects. Attackers can also guess or scrape other users' S3 URLs and attach them to their own tweets.
**The Architectural Fix:** Implement an "Upload Intent" pattern. Create a `Media` table in PostgreSQL. When the client requests an upload, insert a record `status: 'PENDING', userId`. The client uploads to S3. During `createTweet`, the server queries the `Media` table to verify `status === 'PENDING'` and `userId === session.userId`, then updates status to `ATTACHED` within the same transaction. A daily cron job deletes S3 objects and DB records for any `Media` stuck in `PENDING` for over 24 hours.

**Title:** Cache Invalidation Thundering Herd on Soft Deletes
**Category & Severity:** Scalability - High
**The Failure Mode:** Bumping `feed:version:{userId}` on a tweet deletion for _all_ followers of an author is fatal for high-follower accounts. If Taylor Swift deletes a tweet, 100 million follower caches are busted. The next time those users open the app, 100 million concurrent PostgreSQL UNION queries hit the database, instantly triggering connection exhaustion and OOM crashes.
**The Architectural Fix:** Do not bump follower feed versions on delete. Instead, implement "Read-Time Tombstone Filtering." When a tweet is deleted, add its ID to a Redis Set: `tombstones:tweets`. When serving a cached feed page to a user, intersect the cached tweet IDs against the tombstone set. If a match is found, filter it out in memory before returning to the client. The cache naturally rotates out the deleted tweet when its 60s TTL expires, completely avoiding the database herd.

**Title:** Missing Mutation Idempotency
**Category & Severity:** Consistency - High
**The Failure Mode:** Mobile networks drop connections frequently. A user submits a tweet, the server processes it and commits to PostgreSQL, but the TCP connection drops before the 200 OK reaches the phone. The client auto-retries. The server receives the retry and creates a duplicate tweet.
**The Architectural Fix:** Require an `Idempotency-Key` (UUIDv4) header for all mutating endpoints (`createTweet`, `like`, `retweet`). Store this key in Redis `idempotency:{userId}:{key}` with a 24-hour TTL, or in a dedicated PostgreSQL table. If a request arrives with an existing key, bypass the handler and return the cached 200 OK response from the initial successful execution.

**Title:** Redis Pub/Sub vs. Replay Buffer Atomicity
**Category & Severity:** Consistency - Medium
**The Failure Mode:** The SSE design specifies pushing events to Redis Pub/Sub and also to a `sse:replay:{userId}` List. If done in Node.js sequentially, a process crash or network blip between the `PUBLISH` command and the `LPUSH` command means the event is broadcast but not saved for reconnects, or vice versa.
**The Architectural Fix:** Write a custom Lua script loaded into Redis that executes both `PUBLISH` and `LPUSH` (along with `LTRIM` to cap the buffer size) atomically. The Node.js server executes this single script via `redis.eval()`. This guarantees the replay buffer and the live stream are perfectly synchronized at the engine level.

**Title:** In-Memory SETNX Lock Stalling
**Category & Severity:** Availability - Medium
**The Failure Mode:** The `feed:{userId}:rebuilding` SETNX lock prevents simultaneous DB queries. However, if the lock is held, other requests "wait briefly" or return stale data. If the cache is entirely empty (no stale data), waiting requests will block the thread. If the worker holding the lock crashes, all waiting requests hang until the 5s TTL expires, causing massive latency spikes.
**The Architectural Fix:** Use a robust Promise Coalescing pattern combined with a background SWR (Stale-While-Revalidate) worker. If cache misses, the _first_ request creates a Promise that queries the DB and saves it to a local Map. All concurrent requests for that same `userId` await the _same_ Promise instance in memory, rather than hitting Redis/DB. Once resolved, the Promise is removed.

**Title:** Viral Tweet Notification Spikes (The Justin Bieber Problem)
**Category & Severity:** Scalability - High
**The Failure Mode:** A celebrity posts a tweet, and it receives 50,000 likes per minute. The `like` mutation calls `createNotification`, doing 50,000 individual `INSERT` operations into the `Notification` table, destroying database write capacity and starving the SSE publisher.
**The Architectural Fix:** Implement Notification Aggregation for high-velocity entities. Do not insert 1:1 notifications for likes on tweets older than 5 minutes or with over 1,000 likes. Instead, background workers aggregate these and insert a single record: "User A and 4,999 others liked your tweet."

**Title:** OAuth Unique Constraint DB Thrashing
**Category & Severity:** Resilience - Medium
**The Failure Mode:** Relying on `P2002` (Unique Constraint Violation) to handle auto-generated username collisions inside a recursive retry loop is an anti-pattern. If a malicious user registers "john" and bots instantly squat "john_1000" through "john_9999", the retry loop will thrash the database connection pool, repeatedly throwing and catching exceptions until failure.
**The Architectural Fix:** Generate the CUID-based user ID _first_. Append the first 6 characters of the CUID to the sanitized display name to guarantee global uniqueness mathematically on the first attempt (e.g., `johndoe_abc123`). Zero retries, zero database exception handling required.

**Title:** Unbounded Index Scan on Fan-Out-On-Read
**Category & Severity:** Database Performance - High
**The Failure Mode:** The UNION query for the home feed scans `WHERE authorId IN (SELECT followingId...)`. If a user follows 5,000 active accounts, PostgreSQL must scan the indexes for all 5,000 authors, sort the combined results by `createdAt DESC`, and then `LIMIT 21`. As the `Tweet` table grows to billions of rows, this query's execution time will degrade from 10ms to 5 seconds.
**The Architectural Fix:** The Fan-Out-On-Read architecture is fundamentally unsuited for users with massive following graphs. Implement a hybrid architecture. Users following < 500 accounts use Fan-Out-On-Read. Users following > 500 accounts are migrated to Fan-Out-On-Write (their home feeds are pre-materialized in Redis Sorted Sets by background workers).

**Title:** Opaque Cursor Collision Non-Determinism
**Category & Severity:** Consistency - Medium
**The Failure Mode:** The opaque base64 cursor uses `{ ts, id }`. If two tweets are created in the exact same millisecond, and their CUIDs happen to sort in a way that splits across a pagination boundary, items can be duplicated or dropped from the client's perspective during the next fetch.
**The Architectural Fix:** Ensure the `ts` used in the cursor is stored in the database with microsecond precision (`DateTime(6)` in Prisma/Postgres), not standard JS milliseconds. Furthermore, the `id` field must be strictly monotonically sortable. Since standard CUIDs are only _roughly_ time-ordered, use ULIDs or CUID2s for the primary key to ensure exact lexicographical time-ordering.

**Title:** Missing Distributed Tracing Context
**Category & Severity:** Observability - Medium
**The Failure Mode:** When a request fails in production, the logs will show a tRPC error, a Prisma error, and a Redis timeout, but there will be no way to correlate them to a single user action. Debugging concurrent node failures becomes impossible.
**The Architectural Fix:** Implement a tRPC middleware that generates a `x-request-id` (UUIDv4) at the edge. Inject this ID into the tRPC `ctx`. Pass `ctx.reqId` to _every_ database query, Redis call, and logger invocation.

**Title:** SSE Connection Draining & File Descriptor Exhaustion
**Category & Severity:** Availability - High
**The Failure Mode:** The SSE implementation holds HTTP connections open indefinitely. When you deploy a new version of the Next.js server, standard graceful shutdown waits for active connections to close. Since SSE connections never close voluntarily, the deployment hangs until the orchestrator (Kubernetes/Docker) forcefully SIGKILLs the container, dropping all connections abruptly.
**The Architectural Fix:** Implement connection draining. Keep a global `Set` of active SSE response streams. During a `SIGTERM` signal, send a custom `event: server_restart` to all connected clients instructing them to disconnect and jitter-reconnect, then gracefully close the streams from the server side.

**Title:** Unbounded User Input in PostgreSQL ILIKE
**Category & Severity:** Performance / DoS - Medium
**The Failure Mode:** The user search query uses `ILIKE '%' || $1 || '%'`. A malicious user can input a string of 100 wildcard characters (`%%%%...`) or complex regex patterns. While Prisma protects against SQL injection, the pg*trgm engine will choke on extreme wildcard combinations, causing a CPU spike on the database leading to a Denial of Service.
**The Architectural Fix:** Sanitize the search input before it reaches the SQL layer. Strip leading/trailing wildcards, limit the query length to 50 characters, and escape literal `%` and `*` characters in the application logic.

**Title:** Prisma Connection Pooling in Edge/Serverless
**Category & Severity:** Infrastructure - High
**The Failure Mode:** If the application is ever deployed to Vercel or AWS Lambda, the Prisma client will instantiate a new connection pool per isolated function instance. A spike in traffic will instantly exceed the PostgreSQL `max_connections` limit (typically 100), causing cascading `503` errors.
**The Architectural Fix:** Architect the infrastructure to mandate an external connection pooler (e.g., PgBouncer) deployed alongside the database. Do not rely on Prisma's internal pooler for scaling beyond a single monolithic container. Update the database URL configuration to use the PgBouncer port.

**Title:** Stale Client State on Soft Delete
**Category & Severity:** UX Consistency - Low
**The Failure Mode:** When a user deletes a tweet, the server marks it deleted. However, if that tweet is already loaded in the client-side React state of 1,000 other users viewing their feed, the tweet remains visible and interactable. If they attempt to like it, the server returns a 404, causing an unhandled UI error.
**The Architectural Fix:** Broadcast a `tweet_deleted` SSE event to all users holding an active connection. The `use-sse` hook should listen for this event and issue a React Query `setQueryData` imperative update to remove the tweet from all local feed caches instantly.

## 5) Prioritization Roadmap

**1. Must Do Now (Foundational Security & Consistency)**

- Implement BullMQ/Task Queues for Email to close the SMTP timing attack vulnerability.
- Implement the `Media` table Upload Intent flow to stop the orphaned/malicious S3 upload vector.
- Enforce `Idempotency-Key` tracking on mutating endpoints to stop duplicate tweets/likes on network retries.
- Implement RequestId middleware and Context injection for baseline observability.

**2. Should Do Soon (Scaling & Operations)**

- Implement the Tombstone caching pattern instead of Version Bumping for soft deletes to prevent the Cache Invalidation Thundering Herd.
- Write the Lua script for atomic Redis Pub/Sub + Replay Buffer injection to stabilize the SSE real-time feed.
- Refactor the `P2002` OAuth collision loop to use ID-based derivation.
- Implement SSE Connection Draining on `SIGTERM`.

**3. Nice To Have Later (Optimization)**

- Implement Promise Coalescing for cache-miss SETNX lock stalling.
- Migrate high-follower-count users to Fan-Out-On-Write logic.
- Implement background aggregation for viral tweet notifications.
